## 关键错误详解与修复方案

### 错误1: Real.pi_gt_31415 不存在
```lean
-- 原代码 (报错)
have hpi : Real.pi > 3.14 := by linarith [Real.pi_gt_31415]

-- 修复方案1: 使用正确的常量
have hpi : Real.pi > 3.14 := by linarith [Real.pi_gt_314]

-- 修复方案2: 直接使用已知事实
have hpi : Real.pi > 3.14 := by
  have h : Real.pi > 3.14 := Real.pi_gt_314
  linarith
```

**说明**: Mathlib中精确的pi边界常量是`Real.pi_gt_314`，而非`Real.pi_gt_31415`。

### 错误2: intermediate_value_Icc 类型不匹配
```lean
-- 原代码 (报错)
obtain ⟨c, hc_mem, hc_zero⟩ := intermediate_value_Icc hab.le hf h2.1 h2.2

-- 问题分析
-- intermediate_value_Icc 返回类型: ∃ x ∈ Icc a b, f x ∈ Icc (f a) (f b)
-- 但代码期望: ∃ c, c ∈ Set.Icc a b ∧ f c = 0
```

**根本原因**: Mathlib API变更，`intermediate_value_Icc`现在返回的是集合成员关系而非等式。

### 错误3: 重复定义
```lean
-- 修复前: RiemannHypothesis.lean 独立定义
structure ZeroVerification where ...

-- 修复后: 仅导入使用
import SylvaFormalization.ZetaVerifier
-- ZeroVerification 现在通过ZetaVerifier引入
```

## 附录: 截肢降级策略说明

**截肢降级**（Amputation Degradation）是一种Lean形式化工程的应急策略：

> 当复杂证明导致编译失败且短期内无法修复时，将证明体替换为`sorry`占位符，确保模块结构可编译，待后续逐步填充。

**适用条件**:
- 证明涉及复杂的数学定理（IVT、数值分析等）
- Mathlib API变更导致类型不匹配
- 时间紧迫需要优先保证编译通过

**后续回填方法**:
1. 保留原证明作为注释或分支
2. 创建TODO列表追踪每个sorry
3. 使用`#print axioms`检查项目依赖的公理
4. 逐步用实际证明替换sorry
